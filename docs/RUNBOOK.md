# 実験運用手順

対象プロトコル: `R8-010-2x2-screen-v3`

正式刺激は、参加者本人を測定したものではない同一の固定模擬データと、参加者画面内のフグである。心拍等の生体データ、USB機器、空圧装置、物理フグは使用しない。

正式運用は会場のWindows PC 1台、`127.0.0.1`、`device.mode=screen`で行う。外部回答導線、外部ネットワーク、別端末LAN運用は使用しない。

## 1. EXTERNAL COMPLIANCE MODE

正式productionは`compliance.mode=external`で運用する。倫理承認資料と実施条件は研究責任者および当日の運用責任者が本システム外で管理する。本アプリは承認PDF、承認文書参照、承認文書のSHA-256、確認者情報、署名を要求・保存・検証せず、「承認済み」とも表示しない。

Operator画面では次を分けて表示する。

```text
技術状態：実施可能
参加者モード：有効
承認証跡：本システム外で管理
本システムによる承認検証：実施しない
```

旧`goEvidence`、二名照合、screen pilot件数、manual GO ticketは本番release/startのハードゲートではない。技術状態`GO`は倫理承認をアプリが検証したという意味ではない。

## 2. 外部回答との境界

参加者画面とOperatorは、外部回答に関する名称、導線、回答方法、回答完了を表示・確認しない。アプリは外部回答を取得、表示、案内、送信、複製、完了確認しない。

外部調査を使用する場合は、研究スタッフがアプリ外の承認済み手順で別途運用する。提示前同意もアプリ外で取得・記録する。

各提示の`reset`完了後は`response`で自動進行を停止し、参加者画面に次だけを表示する。`{n}`は1〜4の提示番号である。

```text
第{n}提示は終了しました

研究スタッフの案内をお待ちください。
```

スタッフは承認済みのアプリ外運用手順に従って参加者を案内し、待機表示を確認してからOperatorの「参加者への案内を確認して次へ」を押す。APIは`POST /api/sessions/:id/confirm-response-checkpoint`である。第1〜第3提示後は次の`handling`、第4提示後は`summary`へ進む。この確認は外部回答の内容、送信または完了確認ではない。

4提示後のサマリーは、参加者画面に次だけを表示する。

```text
4つの提示は終了しました

4つの提示は以上です。
研究スタッフの案内をお待ちください。
```

スタッフは一般的な引継ぎを行い、Operatorの「スタッフ引継ぎを確認」で完了する。これは外部回答の完了確認ではない。

## 3. 中止の判断

次の場合は直ちに中止する。

- 参加者が中止を申し出た
- 体調不良、不安、画面酔いがある
- 固定値、文言、順序、時間、レイアウトが承認版と異なる
- 画面上のフグが止まらない、またはサーバ状態と一致しない
- Operator接続が失われた
- `result`または`reset`中に参加者画面が切断・再読み込みされた

中止手順:

1. Operatorの「実験を中止」または「緊急停止」を押す。
2. STOP、DEFLATE、画面刺激の停止・収縮を確認する。
3. 参加者画面が中立な中断表示へ移ったことを確認する。
4. `aborted`または`error`がローカルログへ記録されたことを確認する。
5. 参加者の安全を確認し、承認済みの説明・撤回手順へ引き渡す。

`aborted`または`error`のセッションを続きから再開しない。同じ研究用IDを終了状態のまま保持する。

`intro`、`handling`、`processing`、`response`、`summary`中の参加者画面切断は、タイマー停止と中立な復旧表示へ移す。再接続後もスタッフが表示を確認するまで再開しない。`response`は復旧後もOperatorが回答チェックポイントを明示確認するまで進めない。確認できなければ中止する。

## 5. 任意のscreen品質確認

screen pilotは任意の品質確認であり、未実施または件数0でも参加者モードを拒否しない。実施する場合は、研究チームの非参加者だけが必須5コマンドを通過したcleanな候補commitで行う。

1. `npm run screen-pilot`を使う。既存`dist-server/`の直接実行・流用はしない。
2. `127.0.0.1:4174`、`PILOT-xxx`、ScreenPufferDevice、隔離ログを確認する。
3. Operatorに `非参加者用の事前確認` と `画面版・PILOT/テスト`、参加者側に `非参加者用の事前確認` と `外部回答送信なし` が常設されることを確認する。
4. 外部回答導線がないことを確認する。
5. 必要な範囲で4順序、正式時間、固定値、4回の`response`停止・明示確認、6秒膨張・保持・6秒収縮を確認する。
6. source commit等を記録する場合は技術的な再現性確認にだけ使用し、倫理承認証跡として扱わない。

実参加者、正式`SH26-xxx`、外部回答を使用しない。

## 6. 当日の起動

1. 不要なクラウド同期、通知、スリープ、ブラウザ拡張を停止する。
2. PCが外部公開されず、`127.0.0.1`だけを使用することを確認する。
3. 物理フグ、USBシリアル機器、心拍・生体センサが未接続であることを確認する。
4. `VERIFY_RELEASE.cmd`を実行し、manifest、source commit、appVersion、external compliance設定の技術的整合がPASSすることを確認する。
5. `START_PRODUCTION.cmd`を実行する。
6. `CHECK_HEALTH.cmd`が`R8-010-2x2-screen-v3`、`deviceMode=screen`、検証対象の技術設定hashを返すことを確認する。
7. 同じPCで`http://127.0.0.1:4173/operator`と`/device-test`を開く。
8. ScreenPufferDeviceが`idle`、level 0、faultなしであることを確認する。
9. 6秒膨張、保持、6秒収縮、STOP、DEFLATEを確認する。
10. 参加者画面を同じPCの運用対象Chromiumで全画面表示する。
11. Operator画面の「外部管理事項と当日運用の確認」で、当日の手順、提示前同意、実験中止操作、必要時のSTOP・収縮を確認する。
12. 氏名、メール、ID、署名、承認番号、承認文書、SHA-256を入力していないことを確認する。

production起動後に`DEV-xxx`、`DEMO-xxx`、`PILOT-xxx`で練習しない。

## 7. 参加者ごとの手順

1. 提示開始前に、参加者が研究説明を読み、Googleフォームで参加同意を完了したことをアプリ外で確認する。同意しない場合は開始しない。
2. 承認済み管理票の研究用IDだけをOperatorへ入力する。氏名、メール、外部回答情報は入力しない。
3. Operatorの同意確認欄をチェックし、自動割付でセッションを作成する。
4. 専用URLで参加者画面を開き、接続済み、全画面、`screen` ready、idle、level 0を確認する。
5. 共通導入の「固定模擬データ」「本人を測定していない」「生体データ非取得」「比較用クラウドで実送信しない」を読める状態にする。
6. 提示を開始する。各提示の`handling`から`reset`までは残り時間、接続、画面刺激状態だけを監視し、条件ごとの操作をしない。
7. 各提示後の`response`で中立な2文へ置き換わり、自動進行が停止したことを確認する。
8. アプリ外の運用手順で、参加者が今見た提示について11項目へ回答できるよう案内する。アプリへ外部回答情報を入力せず、回答内容・送信・完了をアプリに確認させない。
9. Operatorの「参加者への案内を確認して次へ」を押す。APIは`POST /api/sessions/:id/confirm-response-checkpoint`である。これを4提示それぞれで行う。
10. 第4提示後の確認でサマリーへ進み、固定の終了文言が表示されたら、参加者へ研究スタッフの案内を待ってもらう。
11. スタッフ引継ぎを行う。外部調査がある場合もアプリ外で扱う。
12. Operatorの「スタッフ引継ぎを確認」を押す。APIは`POST /api/sessions/:id/confirm-staff-handoff`である。
13. `completed`を確認して次の参加者へリセットする。

## 8. 切断・停電・異常時

- `result`/`reset`中の参加者画面切断はSTOP、DEFLATE、`error`とし、再開しない。
- `intro`、`handling`、`processing`、`response`、`summary`中の切断は進行を止め、再接続後もスタッフ確認まで再開しない。
- 最後の有効なOperator leaseが進行中に失われた場合は、STOP、DEFLATE、`OPERATOR_CONNECTION_LOST`で`error`へ移す。
- 固定値72以外、承認されていない文言、時間ずれ、C/D差異を見つけたら中止する。
- 停電時は新しい提示を開始せず、進行中セッションを中止する。復旧後も続きから再開しない。
- 未終端ログを削除・書換えない。提示を1回以上開始した順序は使用済み件数へ含める。

## 9. 終了処理

1. 新規・進行中セッションがないことを確認する。
2. STOP、DEFLATE、`idle`、level 0を確認する。
3. サーバ端末でCtrl+Cを1回押し、安全終了とポート閉鎖を確認する。
4. JSONL件数、研究用ID、終了状態を確認し、承認済み暗号化ローカル保存先へ移す。
5. 実ログをGit、チャット、issue、テスト、正式成果物へ含めない。
6. 撤回、分析除外、削除はサーバを停止し、[研究データの撤回・除外・保持期限手順](DATA_LIFECYCLE.md)に従って承認済み外部手順へ引き渡す。

## 10. 各セッションの開始確認

- [ ] 参加者モードが有効
- [ ] Operatorの当日セッション内確認が有効
- [ ] 対象参加者の提示前同意を確認
- [ ] 緊急停止が利用可能
- [ ] 必須runtime checkがPASS

いずれかが欠ける場合は第1提示を開始しない。Operatorの確認状態はサーバメモリまたは`sessionStorage`だけに保持し、アプリまたはブラウザの再起動後は再確認する。この確認は倫理承認の証跡ではない。

## 11. 将来の物理フグ版

`device.mode=serial`、USBシリアル、空圧、物理緊急停止を使う版はscreen-v3に含めない。別protocolVersion、研究計画、倫理判断、同意文、装置安全試験、RUNBOOKを作成する。
