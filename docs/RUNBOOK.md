# 実験運用手順

対象プロトコル: `R8-010-2x2-mock-v1`

## 1. 中止の判断と優先順位

### 物理緊急停止

装置の異音、異臭、過熱、空気漏れ、想定外または止まらない膨張、チューブ抜け、圧迫、STOP/DEFLATE無応答があれば、**画面操作や通信応答を待たず物理緊急停止を最優先**します。

1. 物理緊急停止または装置担当者が定めた空圧・電源の安全停止を実施します。
2. 参加者を装置から離し、安全を確認します。
3. 安全に操作できる場合だけ、アプリの「緊急停止」でSTOP/DEFLATEと中断記録を試みます。
4. 装置を隔離して装置担当者と研究責任者へ連絡します。

異常膨張時にアプリの応答、Ctrl+C、ACK timeoutを待ってはいけません。ソフトウェア表示より物理状態を優先し、原因確認と実機安全試験が完了するまで再通電・再接続しません。

### 通常中止

装置が物理的に正常で、参加者またはスタッフから中止の申し出があった場合や、固定値・提示順・文言・時間が承認版と異なる場合は通常中止です。スタッフ画面の「実験を中止」を使い、STOP/DEFLATEと`aborted`を確認します。通常中止中に装置異常が見つかった場合は直ちに物理緊急停止へ切り替えます。

どちらの中止でも当該セッションは再開しません。研究計画に従って同じ研究用IDを終了状態のまま保持し、必要な再実施は研究責任者の判断で別セッションとして扱います。

## 3. 実機安全確認

装置担当者と2名で確認します。

- [ ] 実機側の最大圧力・最大膨張上限が有効
- [ ] 通信断時に実機単体で安全停止・排気できる
- [ ] 物理緊急停止が動作
- [ ] 配線とチューブに抜け、折れ、圧迫がない
- [ ] 空気漏れがない
- [ ] STOP後の安全状態とDEFLATE受理条件が確認済み
- [ ] 6秒膨張、保持、6秒収縮が承認範囲内
- [ ] 収縮完了をSTATUSまたは装置イベントで確認できる
- [ ] ACK遅延とACK timeoutでSTOP、DEFLATE、中断へ移る
- [ ] USB切断、装置fault、不正応答で安全側へ移る
- [ ] 膨張中の通信断でも実機単体の上限と排気が働く
- [ ] 膨張中の停電・PC電源断でも危険な保持や再膨張が起きない
- [ ] 物理緊急停止後、意図しない自動復帰・自動再接続が起きない
- [ ] 承認回数の連続動作後も発熱、漏れ、タイミングずれがない

`/device-test`を開き、接続→PING→STATUS→上限以下の膨張テスト→保持→収縮→STOPの順に確認します。INFLATE・DEFLATE・STOPでは画面に表示された実ACKの`requestId`、状態、レベル、エラーコードを照合します。正常系だけでなく、装置担当者の安全な故障注入手順でACK timeout、USB切断、fault、電源断も確認します。本番セッション中にデバイステスト画面を操作しません。

## 4. 当日の起動

1. UPS、装置、PC、USB、空圧を承認済みの順序で接続し、物理緊急停止へ直ちに手が届く位置にします。
2. PCの不要なクラウド同期、通知、スリープを停止します。
3. 実験PCと表示端末を隔離LANまたは単一PCに接続し、インターネットへの経路がないことを確認します。Googleフォーム用端末は別経路にします。
4. LANを使う場合は`allowLan`、bind先、OSファイアウォールの許可範囲を2名で照合します。生成された操作トークンをチャット、ログ、写真、スクリーンショットへ残しません。
5. GoogleフォームのURLとQRを2名で開き、承認版の研究説明、同意チェック、11問、個人情報非収集設定を照合し、確認者名と日時を紙または承認済み管理票へ記録します。
6. 配置済みリリースの`VERIFY_RELEASE.cmd`を実行し、全ファイル、Node版、Windowsアーキテクチャ、設定SHA-256が承認記録と一致して`PASS`することを2名で確認します。
7. `START_PRODUCTION.cmd`を実行します。ランチャー内の本番preflightで、Serial、COM、Google Forms、外部通信禁止、ログ先、空き容量がすべて`PASS`した場合だけ起動することを確認します。会場PCではnpm install、build、ソースからの直接起動を行いません。
8. `CHECK_HEALTH.cmd`が`PASS`し、`/healthz`のプロトコル版と`deviceMode=serial`が本番設定に一致することを確認します。
9. `/operator`と参加者用ディスプレイを開きます。
10. 表示をフルスクリーンにし、スクロールや欠けがないことを確認します。
11. 会場照明、表示距離、フォント可読性、フグの設置方向を確認します。
12. 練習用研究IDで1セッション完走後、そのログを実参加者データと分離します。

## 5. 参加者ごとの手順

1. 参加者が自身の端末でGoogleフォームの研究説明を読み、同意チェックを完了したことを確認します。
2. 氏名やメールではなく、割り当て済み研究用IDだけをスタッフ画面へ入力します。
3. 同意確認済みにチェックし、自動割付を使用します。手動選択時は偏り警告を確認します。
4. セッションを作成し、参加者画面URLを専用画面で開きます。
5. 参加者画面を承認済みChromiumのkioskモードまたはF11で全画面表示し、スタッフ画面の状態表示と実画面を目視確認して「全画面表示を確認済み」にチェックします。Fullscreen API表示だけを根拠にしません。
6. 参加者画面が接続済み、装置がreadyかつdeflated、設定検証済みであることを確認します。
7. 共通導入を表示し、参加者が読めたことを確認してから開始します。
8. 自動進行中は通常操作を行いません。残り時間、画面接続、装置状態だけを監視します。
9. サマリー表示後、参加者へGoogleフォームの11問への回答を案内します。
10. 回答完了を口頭で確認します。回答内容はアプリへ入力しません。
11. 「回答完了を確認」でセッションを完了し、次の参加者へリセットします。

## 6. 異常時

### 通常中止（装置の物理状態が正常）

1. スタッフ画面の「実験を中止」を押します。
2. STOP、DEFLATE、参加者画面の中断表示、`aborted`ログを確認します。
3. 参加者へその場で待つよう案内し、研究責任者の手順に従います。
4. 装置状態に異常が見つかったら、以後は物理緊急停止手順へ切り替えます。

### 物理緊急停止（異常膨張、漏れ、異音、異臭、過熱等）

1. **アプリより先に物理緊急停止**を押し、参加者を装置から離します。
2. 装置担当者の手順で空圧・電源を安全に遮断し、収縮状態を直接確認します。
3. 安全に操作できる場合だけ、スタッフ画面の「緊急停止」で記録とSTOP/DEFLATEを試みます。
4. セッションの`error`または`aborted`、参加者画面の中断表示を確認します。確認できなくても物理安全を優先します。
5. 原因調査、修理、実機安全試験、責任者承認が完了するまで再接続・再開しません。

### 参加者画面の更新・切断

- 通常条件ではタイマーが停止し、スタッフ画面に復旧確認が出ます。
- 画面を再接続しただけでは再開しません。表示と残り時間を確認後、「再開」を明示操作します。
- フグ動作中の切断はSTOP/DEFLATEとerrorを優先し、そのセッションを再開しません。

### スタッフ画面の切断・更新

- 最後のスタッフ画面WebSocketが実行中に失われると、無人進行を防ぐためアプリはSTOP/DEFLATEを試み、`OPERATOR_CONNECTION_LOST`でerrorへ移ります。
- スタッフ画面を再接続しても当該セッションは再開しません。装置の物理状態を確認し、安全なら「実験を中止」で中断を確定します。
- ブラウザ更新を予定して行う場合も、必ず提示開始前またはセッション終了後に行います。

### 装置切断・ACKタイムアウト・fault

- アプリはSTOPと可能な場合のDEFLATEを試行します。
- 通信断ではコマンドが届かない可能性があるため、実機側の安全停止と物理状態を必ず確認します。
- セッションをerrorまたはabortedのまま保存し、続きから再開しません。

### UPS作動・停電・PC電源断

- UPSへ切り替わったら新しい提示を開始せず、進行中セッションを通常中止します。装置状態が異常なら物理緊急停止を優先します。
- PCやサーバが停止しても装置コマンドは届かない前提で、実機側の上限、排気、物理停止により安全を確保します。
- 復電後は装置が完全に収縮し、意図せず自動復帰しないことを確認してから再起動します。停止したセッションは続きから再開しません。
- 再起動時に未終端セッション警告が出た場合、そのログを削除・書換えません。CSVでは未完了のまま監査し、提示を1回以上開始した順序は自動割付の使用済み件数へ含まれます。
- UPS残量、発生時刻、装置状態、対象研究用IDを許可された運用記録へ残し、責任者が再開可否を判断します。

## 7. 終了処理

1. 新規セッションがないことを確認します。
2. 装置をDEFLATEし、idle/deflatedを確認します。
3. 通常終了時だけサーバ端末でCtrl+Cを1回押します。端末に安全終了完了が表示され、プロセスが終了してポートが閉じ、装置が収縮したままであることを待って確認します。先に端末ウィンドウを閉じたりプロセスを強制終了したりしません。
4. Ctrl+C後も終了しない、または装置状態が不明・異常な場合は待ち続けず、物理緊急停止手順へ切り替えます。
5. サーバ終了後、装置担当者の手順で装置とUPSの電源を切ります。
6. JSONL件数、研究用ID、終了状態を確認し、必要なCSVを暗号化された承認済み保存先へ移します。
7. ローカルの保持・削除は研究計画の保管期間に従います。
8. 実ログをGit、チャット、issue、テストデータへ添付しません。
