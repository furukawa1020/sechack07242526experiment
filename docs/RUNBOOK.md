# 実験運用手順

対象プロトコル: `R8-010-2x2-screen-v1`

本プロトコルの正式刺激は、固定模擬データと画面上のフグである。心拍等の生体データ、USB機器、空圧装置、物理フグは使用しない。`device.mode=screen`は外部通信も故障注入も行わない正式アダプタであり、開発用`mock`とは別である。

> **現在は人を対象とする本番NO-GOです。** ソフトウェアは実機なし方式へ移行しているが、[Googleフォーム公開内容監査](FORM_AUDIT.md)の所見解消、提示開始前の同意記録方法、画面刺激版の研究計画との整合、二名照合、研究責任者の最終承認が未完了である。これらを完了して本番preflightがPASSするまで参加者へ提示しない。

## 1. 中止の判断

参加者が中止を申し出た場合、体調不良、不安、画面酔い、表示異常、固定値・文言・順序・時間の不一致、画面上のフグが止まらない、スタッフ画面の切断、または`result`/`reset`中の参加者画面切断が起きた場合は直ちに中止する。`intro`、`handling`、`processing`、`summary`中の参加者画面切断だけは自動中止にせず、中立な復旧表示とタイマー停止へ移し、再接続後にスタッフが表示を確認してから残り時間を再開する。復旧を確認できない場合は中止する。

1. スタッフ画面の「実験を中止」または「緊急停止」を押す。
2. 画面上の刺激が停止・収縮し、参加者画面が中立な中断表示へ移ったことを確認する。
3. `aborted`または`error`がローカル監査ログへ記録されたことを確認する。
4. 参加者の安全を確認し、承認済みの説明・撤回手順に従う。

`aborted`または`error`へ移ったセッションは続きから再開しない。同じ研究用IDを終了状態のまま保持し、再実施の可否は研究責任者が判断する。`result`/`reset`以外のフェーズの接続復旧だけは、上記のスタッフ確認後再開手順を使用できる。画面版のSTOP/DEFLATEは論理状態とアニメーションを安全側へ戻す操作であり、物理装置への命令ではない。

## 3. 画面刺激の確認

本番画面には物理フグを接続しない。`/device-test`は画面刺激の論理状態を確認するために使用する。

- [ ] `deviceMode=screen`、`serialPath=""`、`allowMockInProduction=false`
- [ ] ScreenPufferDeviceの論理ready/statusを確認し、開始前は`idle`・level 0・faultなし（USBのPING、STATUS、ACK確認ではない）
- [ ] INFLATEで6秒間、固定レベル0.6まで線形に膨張する
- [ ] 結果提示終了まで同じ大きさで保持する
- [ ] reset開始から6秒間で収縮し、`idle`・level 0へ戻る
- [ ] 継続接続中の描画がサーバ時刻に同期し、resultまたはreset中の再読み込みではSTOP・DEFLATE・errorとなり再開できない
- [ ] CとDでフグのDOM、文言、レベル、開始時刻、膨張・保持・収縮が同一
- [ ] STOP後に刺激が停止し、DEFLATEで収縮状態へ戻る
- [ ] 1366×768と1920×1080で欠け、スクロール、意図しない色差がない
- [ ] USB、シリアル、外部API、外部画像・フォント・CDNへの通信がない

Mockの切断、ACK遅延、timeout、faultは`npm run rehearsal`または自動テストで確認する。本番`screen`アダプタには故障注入機能を持たせず、本番セッション中に`/device-test`を操作しない。

## 4. 当日の起動

1. PCの不要なクラウド同期、通知、スリープ、ブラウザ拡張を停止する。
2. 実験PCと表示端末は単一PCまたは承認済み隔離LANに置く。Googleフォーム回答用端末は別経路とし、実験アプリはフォームを自動取得・送信しない。
3. 物理フグ、USBシリアル機器、心拍・生体センサが接続されていないことを二名で確認する。
4. GoogleフォームのURL、QR、公開payload、管理設定、未ログイン回答経路を二名で再確認する。HTTP 200やタイトル一致だけで済ませない。
5. `VERIFY_RELEASE.cmd`を実行し、全ファイル、Node版、architecture、manifest SHA-256、source commit、設定SHA-256が承認記録と一致してPASSすることを確認する。
6. `START_PRODUCTION.cmd`を実行する。manifest検証と本番preflightの両方がPASSした場合だけサーバを使用する。
7. `CHECK_HEALTH.cmd`がPASSし、`protocolVersion=R8-010-2x2-screen-v1`、`deviceMode=screen`、設定hashが本番設定と一致することを確認する。
8. `/operator`、参加者画面、`/device-test`を開き、前節の画面刺激確認を完了する。
9. 参加者画面を承認済みChromiumのF11またはkioskモードで全画面表示する。全画面APIの通知だけでなく目視でも確認する。
10. Mockリハーサルは別設定・別起動で事前に完走して終了する。production起動後は模擬IDを使用せず、`/device-test`と研究責任者が承認したパイロット手順だけで確認する。

## 5. 参加者ごとの手順

1. 提示開始前に、参加者が研究説明を読み、承認済み方法で参加同意を完了したことを確認する。口頭確認だけで開始しない。
2. フォームまたは承認済み管理票の研究用IDと同じIDだけをスタッフ画面へ入力する。氏名やメールアドレスは入力しない。
3. スタッフ画面の同意確認欄をチェックし、自動割付でセッションを作成する。
4. 専用URLで参加者画面を開き、接続済み、全画面、`screen` ready、idle、level 0、設定検証済みを確認する。
5. 共通導入で「固定模擬データ」「本人を測定していない」「生体データ非取得」「比較用クラウドで実送信しない」を参加者が読めたことを確認する。
6. 提示を開始する。自動進行中は残り時間、画面接続、画面刺激状態だけを監視し、条件ごとの操作をしない。
7. サマリー表示後、同じ研究用IDでGoogleフォームの承認済み11問へ回答するよう案内する。回答内容はスタッフ画面へ入力しない。
8. 回答完了を確認してセッションを完了し、次の参加者へリセットする。

## 6. 切断・停電・異常時

- 参加者画面の切断では進行を止める。フグ表示中の切断はSTOP、DEFLATE、`error`を優先し、そのセッションを再開しない。
- 最後のスタッフ画面WebSocketが進行中に失われた場合は、無人進行を防ぐためSTOP、DEFLATE、`OPERATOR_CONNECTION_LOST`で`error`へ移す。
- 画面上のフグがサーバ状態と一致しない、固定値72以外を示す、表示時間がずれる、文言が承認版と違う場合は中止する。
- UPS作動または停電時は新しい提示を開始しない。進行中セッションを中止し、復旧後も続きから再開しない。
- 未終端セッションのログは削除・書換えない。提示を1回以上開始した順序は割付の使用済み件数へ含める。

## 7. 終了処理

1. 新規・進行中セッションがないことを確認する。
2. 画面刺激をSTOP、DEFLATEし、idle・level 0を確認する。
3. サーバ端末でCtrl+Cを1回押し、安全終了、ポート閉鎖を確認する。ウィンドウを先に閉じたり強制終了したりしない。
4. JSONL件数、研究用ID、終了状態を確認し、必要なCSVを承認済みの暗号化ローカル保存先へ移す。
5. 保持・削除期間は研究計画に従う。実ログをGit、チャット、issue、テストへ含めない。

## 9. 将来の物理フグ版

`device.mode=serial`、USBシリアル、空圧、物理緊急停止を使用する版は本プロトコルに含めない。実施する場合は別の`protocolVersion`、研究計画、同意文、装置安全試験、RUNBOOKを作成し、本画面版のGO記録を流用しない。
