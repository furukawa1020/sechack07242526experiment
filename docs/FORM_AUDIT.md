## 1. 監査範囲

公開URLへのHTTP応答と、公開応答に含まれる文字列を機械的に確認した。Google Formsの編集者画面、回答保存先、アカウント制限、メールアドレス収集設定等の管理者専用設定は監査対象外である。

「公開payload」は、参加者がアクセスできる公開応答に含まれるデータを指す。通常画面で常時表示される文字列だけを意味しない。

## 4. リスク評価

次は監査事実から導く運用上の判定であり、機械抽出された事実そのものではない。

- 内部コードと固定対応を公開payloadへ含めることは、参加者へA/B/C/Dを見せない研究条件に反し、仮説推測や回答バイアスを招くおそれがある。
- 現在のタイトルは事後評価フォームを同意取得経路と誤認させ得る。固定タイトル`身体状態の外化デバイスがユーザの心理状態に及ぼす影響の評価｜提示後アンケート`へ完全一致させ、研究説明と同意は別の承認済み経路で提示前に提供・記録する必要がある。
- 3種類と4種類の併存、および回答タイミングの併存は、参加者への説明と実際の実験手順を一致させられない。
- screen版の必須説明が0件のため、参加者が実データや物理フグを使う研究だと誤解するおそれがあり、`R8-010-2x2-screen-v1`の研究計画・同意・実装と一致しない。
- 無題の回答入力項目は参加者が何を回答すべきか判断できず、意図しない回答または分岐を生じさせる可能性がある。
- 研究用ID欄の固定ラベルと形式validationがないため、ローカルログと結合できない入力や、氏名等の誤入力を機械的に防げない。
- 提示順コードを参加者へ入力させる説明は、参加者へ内部割付情報を見せない条件と矛盾する。
- 11評価項目は確認できたが、内部コード公開、提示数、回答タイミングの不整合が残るため、質問数の一致だけでは本番利用を承認できない。
- 回答項目の種類をPIIキーワードだけで判定すると、年齢区分、同意チェック、無害に見える選択肢等を追加収集できてしまう。事後評価フォームは、厳密な研究用ID欄1件と承認候補構造の評価グリッド11件だけに限定する。
- メール収集設定が未確認の状態では、個人情報を収集しない要件を満たしたと判断できない。

以上のため、フォーム内容は現状のまま本番利用できない。

### 4.1 機械可読の本番ゲート

実験設定の`formAudit`は、フォーム回答や確認者の氏名を保存する欄ではなく、承認済み監査証跡を本番設定へ結び付けるフェイルクローズの記録である。現在の設定は次を明示している。

```json
{
  "status": "NO-GO",
  "protocolVersion": "R8-010-2x2-screen-v1",
  "formUrl": "https://forms.gle/BeShY7cY5zMjunto9",
  "auditedOn": "2026-07-21",
  "contentSha256": "33762250e42e9cb63900ccd58a64923f4047693086a81a3737cfe7cbb72d9476",
  "twoPersonVerified": false
}
```

本番preflightと本番サーバ起動は、次のすべてを満たす場合だけ通過する。

- `status`が`GO`
- 監査記録の`protocolVersion`と`formUrl`が、同じ設定ファイルの値と一言一句一致する
- `formUrl`が指定研究フォームURLと完全一致する
- `twoPersonVerified`が`true`
- `auditedOn`が未来日でなく、実行日から7日以内
- `contentSha256`が、動的なHTML外枠ではなく公開応答内の安定した`FB_PUBLIC_LOAD_DATA_` payloadに対する小文字16進64文字のSHA-256
- `contentSha256`が、既知のNO-GO公開payload SHA-256ではない

1項目でも欠落、不一致、期限切れなら本番は失敗する。`--allow-mock`では警告として開発を継続できるが、本番承認にはならない。確認者の氏名等は研究計画で定めた管理場所へ保管し、設定、ログ、リリースmanifestには入れない。

`npm run audit:form`は公開内容を読取り専用で診断し、固定フォームID、提示後アンケート専用の固定タイトルへの完全一致、公開payload、内部条件、提示数、固定模擬データ・本人非測定・生体データ非取得・画面上フグ・Googleフォーム回答の送信先・実験アプリによる固定模擬データの外部非送信および非保存という必須6点、回答タイミング、実際の11評価質問、第1〜第4提示の行、7件法、任意回答、研究用ID欄の件数・固定ラベル・必須設定・形式validation、回答項目が厳密な研究用ID欄1件と評価グリッド11件の計12件だけであること、提示順・内部コード入力の不存在、禁止された個人情報入力の不存在、研究用ID以外の短文・段落自由記述の不存在、無題入力、ファイルアップロード、および安定した公開payloadのSHA-256を確認する。追加のラジオボタン、プルダウン、チェックボックス、尺度、日付、時刻、ファイルアップロード、未知の回答形式は、文言にPIIキーワードがなくてもNO-GOとする。監査URLとリダイレクト後URLはqueryなし、または`usp=send_form`だけを許可し、prefill用`entry.*`、`usp=pp_url`、未知のquery、fragmentを拒否する。設定を自動更新もGO化もしない。診断後、人手の二名照合と研究責任者の承認を完了してから、同じ公開payloadのSHA-256と日付を設定へ転記する。

`formAudit`がGOでも、それだけではproductionを開始できない。研究計画、倫理判断、提示前同意、データ管理、screenパイロット、独立二名照合は[本番GO証跡手順](GO_EVIDENCE.md)に従って別の`goEvidence`へ結び付ける。
