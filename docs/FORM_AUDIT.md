## 4. リスク評価

次は監査事実から導く運用上の判定であり、機械抽出された事実そのものではない。

- 内部コードと固定対応を公開payloadへ含めることは、参加者へA/B/C/Dを見せない研究条件に反し、仮説推測や回答バイアスを招くおそれがある。
- 3種類と4種類の併存、および回答タイミングの併存は、参加者への説明と実際の実験手順を一致させられない。
- screen版の必須説明が0件のため、参加者が実データや物理フグを使う研究だと誤解するおそれがあり、`R8-010-2x2-screen-v1`の研究計画・同意・実装と一致しない。
- 無題の回答入力項目は参加者が何を回答すべきか判断できず、意図しない回答または分岐を生じさせる可能性がある。
- 11評価項目は確認できたが、内部コード公開、提示数、回答タイミングの不整合が残るため、質問数の一致だけでは本番利用を承認できない。

以上のため、フォーム内容は現状のまま本番利用できない。

### 4.1 機械可読の本番ゲート

実験設定の`formAudit`は、フォーム回答や確認者の氏名を保存する欄ではなく、承認済み監査証跡を本番設定へ結び付けるフェイルクローズの記録である。現在の設定は次を明示している。

```json
{
  "status": "NO-GO",
  "protocolVersion": "R8-010-2x2-screen-v1",
  "formUrl": "https://forms.gle/BeShY7cY5zMjunto9",
  "auditedOn": "2026-07-21",
  "contentSha256": "33762250e42e9cb63900ccd58a64923f4047693086a81a3737cfe7cbb72d9476",
  "twoPersonVerified": false
}
```

本番preflightと本番サーバ起動は、次のすべてを満たす場合だけ通過する。

- `status`が`GO`
- 監査記録の`protocolVersion`と`formUrl`が、同じ設定ファイルの値と一言一句一致する
- `formUrl`が指定研究フォームURLと完全一致する
- `twoPersonVerified`が`true`
- `auditedOn`が未来日でなく、実行日から7日以内
- `contentSha256`が、動的なHTML外枠ではなく公開応答内の安定した`FB_PUBLIC_LOAD_DATA_` payloadに対する小文字16進64文字のSHA-256
- `contentSha256`が、既知のNO-GO公開payload SHA-256ではない

1項目でも欠落、不一致、期限切れなら本番は失敗する。`--allow-mock`では警告として開発を継続できるが、本番承認にはならない。確認者の氏名等は研究計画で定めた管理場所へ保管し、設定、ログ、リリースmanifestには入れない。

`npm run audit:form`は公開内容を読取り専用で診断し、固定フォームID、研究タイトル、公開payload、内部条件、提示数、固定模擬データ・本人非測定・生体データ非取得・画面上フグ・Googleフォーム回答の送信先・実験アプリによる固定模擬データの外部非送信および非保存という必須6点、回答タイミング、実際の11評価質問、第1〜第4提示の行、7件法、任意回答、無題入力、ファイルアップロード、および安定した公開payloadのSHA-256を確認する。設定を自動更新もGO化もしない。診断後、人手の二名照合と研究責任者の承認を完了してから、同じ公開payloadのSHA-256と日付を設定へ転記する。

## 5. NO-GO解除条件

フォーム所有者が承認済み研究計画に基づいて修正した後、次を二名が独立に確認する。

- [ ] 公開画面と公開payloadのどちらにも、内部コードA/B/C/Dおよび固定対応が含まれない
- [ ] 提示数の説明が4提示に統一され、3種類という旧説明が残っていない
- [ ] 回答手順が、4提示とサマリーの終了後にGoogleフォームへ戻って回答する承認済みフローへ統一されている
- [ ] 第1提示から第4提示までを評価する11問が、実際の回答経路で過不足なく表示・回答できる
- [ ] 11問が第1〜第4提示の行、承認済み7件法、任意回答で統一されている
- [ ] タイトルまたは質問文が空の回答入力項目と、ファイルアップロード項目がない
- [ ] 年齢を自由記述で必須回答させる項目がない。年齢情報が承認済み計画で必要な場合も、承認済みの任意・粗い区分と「回答しない」だけを用いる
- [ ] 研究説明、同意チェック、11問および送信完了画面が承認版と一言一句一致する
- [ ] 表示値が参加者本人の測定値ではなく、心拍等の生体データを取得しない固定模擬データ方式であることが明記されている
- [ ] フグ条件が物理装置ではなく画面上の表現であることが明記されている
- [ ] 実験用Webアプリは固定模擬身体データを外部送信しない一方、アンケート回答は参加者がGoogleフォームを送信したときにGoogleへ送信・保存されることを区別して明記している
- [ ] Google Forms管理画面でメールアドレス収集が無効であることを確認する
- [ ] ログイン要求、氏名、メールアドレスその他の個人情報収集がないことを、未ログイン状態の実回答経路でも確認する
- [ ] QRとURLが同じ対象フォームを開く
- [ ] iPhoneとAndroidの両方で、分岐を含む全回答経路を完走する
- [ ] 修正後の公開payloadを再監査し、本書の未解決所見が0件になったことを記録する
- [ ] 確認者2名の氏名、日時、確認したフォーム版または承認済み証跡を研究計画で定めた場所へ記録する

参加者向け説明または回答手順を変更する場合は、研究責任者が研究計画との整合を確認し、必要に応じて`protocolVersion`と`docs/PROTOCOL_CHANGELOG.md`を更新する。本書のチェックだけを先に完了扱いにしてはならない。
