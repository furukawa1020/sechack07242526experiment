# 研究データの撤回・除外・保持期限手順

対象は、正式productionがローカルへ保存した`data/sessions/YYYY-MM-DD/SH26-xxx_<session-id>.jsonl`である。Googleフォーム回答は取得、表示、複製、変更しない。

## 安全上の結論

このアプリと正式リリースには、JSONLを除外・変更・削除する機能を同梱しない。Node.jsだけでは、検証済みの同一FileHandleに結び付いた不可逆なunlinkを全対応OSで保証できないためである。ソース内の除外・削除APIも常にfail-closedで拒否する。

撤回、分析除外、削除の依頼を受けた場合は、実験サーバを停止し、研究責任者が事前承認した外部のデータ管理手順へ引き渡す。承認済み手順がない状態でファイル操作を行ってはならない。

外部手順には最低限、次を含める。

- 研究責任者による依頼内容と対象研究用IDの確認
- 対象JSONLとSHA-256の二名照合
- シンボリックリンク、ジャンクション、複数ハードリンク、`data/sessions`外を拒否する仕組み
- 操作者、承認者、日時、対象、実施結果を残す独立監査記録
- 失敗または一部完了時に再実行せず、責任者へ報告する手順
- Googleフォーム側の同一研究用IDを、権限を持つフォーム管理者が別途照合する手順
- 研究用IDの永続予約registryを保持し、ログの移動・削除後もIDを再利用しない手順

永続予約は`data/research-id-registry/`、初期化anchorは`data/.research-id-registry-initialized.json`に置く。片方だけが消失した場合、アプリは新規ID割付をfail-closedで停止する。`data/`親、anchor、registryは承認済みバックアップ手順でセッションJSONLとは独立に保全する。

ただしanchorとregistryの両方が同時に失われると、ソフトウェアだけでは「未初期化」と区別できない。この場合は過去IDが再利用される残存リスクがあるため、外部の割付台帳と独立バックアップを照合し、研究責任者が復旧を承認するまで実験を再開しない。コードだけで完全な永続性を保証すると扱ってはならない。

## 読取り専用の確認

開発リポジトリでは、候補確認と保持期限レポートだけを読取り専用で実行できる。これらは正式リリースへ同梱されず、ファイルを変更しない。

候補確認:

```powershell
npm.cmd run data:lifecycle -- preview --action delete --research-id SH26-001
```

出力された`targetCount`、相対パス、session ID、サイズ、更新時刻、SHA-256、`planId`は参考情報であり、削除権限や承認を意味しない。アプリ内に、この`planId`を使って変更するコマンドはない。

保持期限候補:

```powershell
npm.cmd run data:lifecycle -- retention-report --retention-days 365
```

基準日を固定する場合:

```powershell
npm.cmd run data:lifecycle -- retention-report --retention-days 365 --as-of 2027-07-24
```

レポートは候補を示すだけで、自動除外・自動削除を行わない。保持日数は承認済みデータ管理計画の値を指定する。

日付基準はUTCへ統一する。セッション保存先の`YYYY-MM-DD`、`collectedOn`、`asOf`の既定値、保持日数の加算はすべてUTC日付であり、JST日付として読み替えない。

## Googleフォーム側の対応

ローカル側の処理だけでGoogleフォーム回答は変わらない。

1. 承認済み権限を持つフォーム管理者が、同じ研究用IDの回答を照合する。
2. データ管理計画に従って、分析対象外化または削除を行う。
3. 実施者、日時、対象ID、ローカル対象のSHA-256、Googleフォーム側の結果を承認済み管理票へ記録する。
4. 氏名、メール、回答内容をアプリ設定、Git、issue、チャット、スクリーンショットへ転記しない。

ローカルとGoogleフォームのどちらか一方だけが未処理なら、撤回・除外・削除を完了扱いにしない。
