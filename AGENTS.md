# AGENTS.md

## 目的

このリポジトリは、SecHack365合宿で実施する人対象研究の実験提示用ローカルWebアプリである。通常のプロダクト開発よりも、次を優先する。

1. 実験条件以外の差を最小化する
2. 参加者へ誤解を与えない
3. 研究データを外部送信しない
4. 装置を安全に停止できる
5. 条件、文言、タイミングを再現可能にする
6. 何が起きたかを研究用ID単位で監査できる

## 必ず読む文書

変更前に次を読むこと。

- `docs/EXPERIMENT_SPEC.md`
- `docs/UI_COPY.md`
- `docs/DEVICE_PROTOCOL.md`
- `config/experiment.json` または `config/experiment.example.json`

矛盾がある場合の優先順位は次のとおり。

1. `AGENTS.md`
2. `docs/EXPERIMENT_SPEC.md`
3. `docs/UI_COPY.md`
4. `docs/DEVICE_PROTOCOL.md`
5. 設定ファイル
6. README

## 研究条件を変更してはならない

固定対応：

- A = cloud + label
- B = local + label
- C = local + puffer
- D = cloud + puffer

参加者画面にA/B/C/Dを表示してはならない。

提示順：

- ABDC
- BCAD
- CDBA
- DACB

条件の色、背景、余白、タイポグラフィ、提示時間を変えて、条件の印象を余計に操作してはならない。

## データ保護

禁止：

- 外部APIへの研究データ送信
- Firebase、Google Analytics、Sentry等のテレメトリ
- 外部CDN、外部フォント、外部画像
- 氏名、メールアドレス、学籍番号、住所、電話番号の収集
- IPアドレス、位置情報、カメラ、マイク、ブラウザ指紋の記録
- Googleフォーム回答の取得・複製
- 実参加者データをfixtureやテストへコピー
- `data/`配下の実ログをGitへ追加

許可されるログ項目は、仕様書で明示された研究用ID、提示順、条件、時刻、固定値、装置イベント、終了状態のみ。

## クラウド条件について

「クラウド」は参加者へ提示する比較用シナリオである。実装内部でクラウド条件だけ外部通信してはならない。cloud/localの分岐は、参加者へ表示する文言と条件メタデータに限定する。

ローカルサーバ自身の通信は `localhost` または明示的に許可された同一LAN内に限定する。フォームURLは、参加者が実験終了後に明示的に開くリンクとしてのみ扱い、アプリから自動取得・送信しない。

## UIの原則

- 参加者画面の文言は `docs/UI_COPY.md` から参照する
- 文言をコンポーネントへ散在させない
- cloud/localを赤/緑で区別しない
- 安全/危険、推奨/非推奨を示すアイコンや言葉を使わない
- ラベル条件A/Bの右側表示は同一
- フグ条件C/Dの右側表示と装置命令は同一
- UI上の差分をテスト可能なデータ構造にする
- 参加者画面に操作ボタンを置かない
- 仮説や期待する結果をアンケート送信前に表示しない
- `この表示は医療上の診断ではありません。`を全結果画面に表示する
- `比較用シナリオ`であることを共通導入と画面フッターで明示する

## 実験状態

サーバを唯一の状態源とする。クライアント固有のタイマーだけで実験を進めてはならない。

最低限の状態：

- idle
- setup
- intro
- handling
- processing
- result
- reset
- summary
- completed
- aborted
- error

不正な遷移を拒否する。すべての遷移をログに残す。

## 装置

装置制御はインターフェースで分離する。

- `MockPufferDevice`
- `SerialPufferDevice`

装置異常、タイムアウト、切断では、次を順に試行する。

1. STOP
2. DEFLATE
3. セッションをerrorまたはabortedへ
4. 参加者画面を中立な中断画面へ
5. ログへ記録

ソフトウェアから安全上限を解除する機能を実装してはならない。圧力値を直接指定せず、0.0〜1.0の正規化レベルだけを送る。実機側が必ず上限をクランプする前提とする。

## コード品質

- TypeScript `strict: true`
- ドメイン層で`any`を使用しない
- 条件定義、文言、タイミングを一元管理
- 設定をZod等で起動時検証
- 外部入力を検証
- 例外を握りつぶさない
- 時刻はISO 8601と単調増加時間を併記
- マジックナンバーを避ける
- 依存を最小限にする
- 外部リソースなしでビルド・起動後の実験画面が動くこと

## テスト

変更後、必ず実行する。

```bash
npm run lint
npm run typecheck
npm test
npm run test:e2e
npm run build
```

失敗したテストを無視、skip、削除して通してはならない。

最低限のカバレッジ：

- 条件マッピング：100%
- 順序割付：100%
- ステートマシン：90%以上
- 装置アダプタ：90%以上
- ログの許可フィールド検証：100%

E2Eでは全4順序をMockDeviceで完走する。

## 変更管理

参加者向け文言、条件定義、タイミング、固定スコア、フグ動作を変更した場合は、必ず次を行う。

- `protocolVersion`を更新
- `docs/PROTOCOL_CHANGELOG.md`へ追記
- 関連テストを更新
- 変更理由を最終報告へ記載

## 完了報告

完了時は次を報告する。

- 実装したもの
- 変更したファイル
- 実行したテストと結果
- 未解決事項
- 実機確認が必要な項目
- データ保護上の確認
- 本番前の運用チェック
