# AGENTS.md

## 目的

このリポジトリは、SecHack365合宿で実施する人対象研究の実験提示用ローカルWebアプリである。通常のプロダクト開発よりも、次を優先する。

1. 実験条件以外の差を最小化する
2. 参加者へ誤解を与えない
3. 研究データを外部送信しない
4. 装置を安全に停止できる
5. 条件、文言、タイミングを再現可能にする
6. 何が起きたかを研究用ID単位で監査できる

## 必ず読む文書

変更前に次を読むこと。

- `docs/EXPERIMENT_SPEC.md`
- `docs/UI_COPY.md`
- `docs/DEVICE_PROTOCOL.md`
- `config/experiment.json` または `config/experiment.example.json`

矛盾がある場合の優先順位は次のとおり。

1. `AGENTS.md`
2. `docs/EXPERIMENT_SPEC.md`
3. `docs/UI_COPY.md`
4. `docs/DEVICE_PROTOCOL.md`
5. 設定ファイル
6. README

## 研究条件を変更してはならない

正式MVPのプロトコルは`R8-010-2x2-screen-v2`とする。この版の入力は、参加者本人を測定したものではない同一の固定模擬データだけであり、心拍その他の生体データを取得しない。伝え方の`puffer`は画面上のフグのふくらみを指す。

固定対応：

- A = cloud + label
- B = local + label
- C = local + puffer
- D = cloud + puffer

参加者画面にA/B/C/Dを表示してはならない。

提示順：

- ABDC
- BCAD
- CDBA
- DACB

条件の色、背景、余白、タイポグラフィ、提示時間を変えて、条件の印象を余計に操作してはならない。

正式MVPでは`device.mode=screen`を使用する。`mock`は開発・自動テスト・明示的な模擬リハーサル専用であり、正式実施に使用してはならない。`serial`による物理フグは将来の別プロトコルであり、`R8-010-2x2-screen-v2`へ混在させてはならない。

## データ保護

禁止：

- 外部APIへの研究データ送信
- 外部分析・エラー監視・テレメトリサービス
- 外部CDN、外部フォント、外部画像
- 氏名、メールアドレス、学籍番号、住所、電話番号の収集
- IPアドレス、位置情報、カメラ、マイク、ブラウザ指紋の記録
- フォームその他の外部アンケート回答の取得・表示・複製・完了確認
- 正式参加者UIへの外部回答に関する名称、導線、回答誘導の表示
- 実参加者データをfixtureやテストへコピー
- `data/`配下の実ログをGitへ追加

許可されるログ項目は、仕様書で明示された研究用ID、提示順、条件、時刻、固定値、装置イベント、終了状態のみ。

## クラウド条件について

「クラウド」は参加者へ提示する比較用シナリオである。実装内部でクラウド条件だけ外部通信してはならない。cloud/localの分岐は、参加者へ表示する文言と条件メタデータに限定する。

正式productionは会場Windows PC 1台の`127.0.0.1`だけで動作させ、外部runtime通信を0件とする。`formUrl`は空、`formAudit`は不在とし、フォームその他の外部アンケートに関する告知・運用は研究スタッフがアプリ外で行う。アプリは外部アンケートを取得、表示、誘導、送信、完了確認しない。

## UIの原則

- 参加者画面の文言は `docs/UI_COPY.md` から参照する
- 文言をコンポーネントへ散在させない
- cloud/localを赤/緑で区別しない
- 安全/危険、推奨/非推奨を示すアイコンや言葉を使わない
- ラベル条件A/Bの右側表示は同一
- フグ条件C/Dの右側表示、画面上のフグの描画、状態遷移および装置抽象化への命令は同一
- UI上の差分をテスト可能なデータ構造にする
- 参加者画面に操作ボタンを置かない
- 仮説や期待する結果を参加者向け画面へ表示しない
- サマリー見出しは`4つの提示は終了しました`、本文は`4つの提示は以上です。\n研究スタッフの案内をお待ちください。`に固定する
- 4提示後の完了操作は汎用的なスタッフ引継ぎ確認とし、API名を`confirm-staff-handoff`に固定する。外部回答の完了確認として扱わない
- `この表示は医療上の診断ではありません。`を全結果画面に表示する
- `比較用シナリオ`であることを共通導入と画面フッターで明示する

## 実験状態

サーバを唯一の状態源とする。クライアント固有のタイマーだけで実験を進めてはならない。

最低限の状態：

- idle
- setup
- intro
- handling
- processing
- result
- reset
- summary
- completed
- aborted
- error

不正な遷移を拒否する。すべての遷移をログに残す。

正式productionリリースには`FORM_*`文書・機能、`MOCK_REHEARSAL`文書・起動物、`PUBLIC_DEMO`文書・成果物を含めない。これらはv1履歴または非参加者向け任意資料であり、v2のrelease/start gateではない。`goEvidence`の研究計画、倫理判断、提示前同意、データ管理計画、非参加者screen pilot 3〜5件、独立二名reviewは引き続きhard fail-closedの必須条件とする。

## 装置

装置制御はインターフェースで分離する。

- `ScreenPufferDevice`
- `MockPufferDevice`
- `SerialPufferDevice`

正式MVPの`ScreenPufferDevice`はUSB機器を必要とせず、外部通信を行わない。`result`開始から6,000msで画面上のフグを膨張させ、結果提示終了まで保持し、`reset`開始から6,000msで収縮させる。描画はサーバ時刻を唯一の基準とし、継続接続中の再描画では同じ時点の形状を復元する。CとDでこの動作を完全に同一にする。`result`または`reset`中の再読み込み・切断は刺激欠損として即時STOP、DEFLATE、errorとし、そのセッションを再開しない。他フェーズの再読み込みは進行を一時停止し、再接続後もOperatorの明示確認なしに再開しない。

以下のSTOP/DEFLATEと物理安全上限の規則は、将来の`SerialPufferDevice`版を実装・審査するときにも維持する。画面版であっても中断時は状態機械上のSTOP、DEFLATEを実行し、中立な中断画面へ移す。

装置異常、タイムアウト、切断では、次を順に試行する。

1. STOP
2. DEFLATE
3. セッションをerrorまたはabortedへ
4. 参加者画面を中立な中断画面へ
5. ログへ記録

ソフトウェアから安全上限を解除する機能を実装してはならない。圧力値を直接指定せず、0.0〜1.0の正規化レベルだけを送る。実機側が必ず上限をクランプする前提とする。

## コード品質

- TypeScript `strict: true`
- ドメイン層で`any`を使用しない
- 条件定義、文言、タイミングを一元管理
- 設定をZod等で起動時検証
- 外部入力を検証
- 例外を握りつぶさない
- 時刻はISO 8601と単調増加時間を併記
- マジックナンバーを避ける
- 依存を最小限にする
- 外部リソースなしでビルド・起動後の実験画面が動くこと

## テスト

変更後、必ず実行する。

```bash
npm run lint
npm run typecheck
npm test
npm run test:e2e
npm run build
```

失敗したテストを無視、skip、削除して通してはならない。

最低限のカバレッジ：

- 条件マッピング：100%
- 順序割付：100%
- ステートマシン：90%以上
- 装置アダプタ：90%以上
- ログの許可フィールド検証：100%

E2Eでは全4順序をMockDeviceで完走する。

加えて、正式な`screen`モードが実機なしで起動できること、C/Dの画面上フグが同一であること、6,000msの膨張・保持・6,000msの収縮がサーバ時刻に同期すること、通常フェーズの再読み込みでOperator確認まで停止すること、`result`または`reset`中の再読み込みで安全停止して再開不能になることをテストする。

## 変更管理

参加者向け文言、条件定義、タイミング、固定スコア、フグ動作を変更した場合は、必ず次を行う。

- `protocolVersion`を更新
- `docs/PROTOCOL_CHANGELOG.md`へ追記
- 関連テストを更新
- 変更理由を最終報告へ記載

物理フグと画面上フグの切替は研究刺激の変更である。実施前に研究責任者の承認を得て、所属機関で必要とされる倫理審査・変更手続きを完了する。承認前に実参加者へ使用してはならない。

## 完了報告

完了時は次を報告する。

- 実装したもの
- 変更したファイル
- 実行したテストと結果
- 未解決事項
- 実機確認が必要な項目
- データ保護上の確認
- 本番前の運用チェック
